# Findings

- The v4 spec mandates that legality is structural, not a post-filter around free network output.
- The atomic action object is `Loadout`, not a bare hero ID. It must include hero, unique equipment ID, unique equipment star, normal equipment summary/features, final power, and standing rank.
- Unique equipment uniqueness is keyed by equipment ID, not by equipment star.
- Team slot order is position-aware and must be strictly increasing by `standing_rank`.
- The minimum deliverable is data structures, legal generator, SingleTeamWinrateModel interface, complete-defense AttackOracle, simulation cache, successive halving, diversity selection, and output explanation.
- Neural networks should not be trained first. Oracle search and legal data generation come before distillation.
- The existing oracle backend already exposes the needed decoupled API: `POST /jobs`, `GET /jobs/{job_id}`, `GET /jobs/{job_id}/results`, and `GET /api/status`.
- In backend result labels, `battle_result == 0` maps to attack win and `battle_result == 1` maps to attack loss; `battle_result == 3` should be treated as a mixed/soft result when unit state is available.
- Real smoke submission should require an already-ready persistent worker pool. A stopped backend is not enough for this repository because worker lifecycle ownership stays outside the algorithm package.
- The decoded `heroes.json` `equipIds` field contains ordinary item IDs such as `875` and `422`. These must not be inferred as unique legend equipment.
- New battle requests must match the old random-lineup proto shape closely. Several heroes can fail in Lua when `_legend_equip`, `_shard`, or `_astrolabe` runtime fields are missing.
- Resource selection for league rounds should use `PeakArenaCampGroup`/`PeakArenaCampList` rather than the full decoded hero catalog.
- Old random-lineup requests assign all unique legend equips once per side, then fill remaining slots with normal legend equips. The league runner now mirrors that side-level rule.
- Full v4 conformance is not the same as the current MVP. The main runtime still uses random defenses plus complete-defense attack search, while the spec requires DefenseOracle roster/mask generation and mask-aware AttackOracle through Observation/BeliefEngine.
- `DefenseOracle` currently discards `attack_meta`; this violates the anti-meta and PSRO sections.
- The neural modules are not implemented yet. `LegalMaskedAttackGenerator` is a legal heuristic proposal generator, not the documented causal AttackGenerationNetwork.
- `LeagueManager` currently stores pools/payoffs and strength-weighted distributions only. It does not yet implement the documented mixed meta distribution, main/exploiter/underdog roles, or historical retention policy.
- Real oracle debugging found hero/loadouts `45` (`小鹿`, script `aEnch`) and `75` (`天怒法师`, script `aSM`) reliably trigger `datatable.lua:22: bad argument #1 to 'f' (table expected, got nil)` in the current backend request shape. `run_league_round.py` now excludes `DEFAULT_ORACLE_EXCLUDED_HERO_IDS=(45, 75)` unless explicitly overridden.
- BeliefEngine real-distribution weighting used to rely on `RealMetaDB.by_observation_hash()` only. It now also retrieves similar real observations by visible slot overlap, visible hero/equip overlap, hidden-slot overlap, season, rank segment, and defense-observation compatibility; `candidates.jsonl` persists `belief_domain_stats` so exact/similar real-meta usage is visible in artifacts.
- `MaskSearcher.search(..., max_masks=1)` previously evaluated the first lexicographic legal mask, which was all-visible and produced `hidden_count=0` in fast live smokes. Team mask patterns now enumerate high-hidden masks first, so bounded searches still exercise the hidden-observation path.
- Belief real-distribution validation is now an explicit scheduled report, not an ad hoc artifact inspection: `belief_real_distribution_validation_report.v1` checks `belief_domain_stats` coverage, real exact/similar evidence, real similarity, and oracle-alignment MAE, with scheduler red lines for missing or low-quality real-distribution usage.
- Defense anti-meta validation is now separate from attack exploiter validation: `defense_anti_meta_effectiveness_report.v1` reads `defense_teacher.jsonl`, checks anti-meta feedback coverage and residual quality, tracks per-round residual trend, and participates in recurring scheduler red-line stops.
- AttackOracle failure annotations now have their own validation gate: `attack_oracle_failure_validation_report.v1` scans direct `attack_oracle_output.v1` JSON and round `candidates.jsonl`, checking `failure_code`, `failure_stage`, matching diagnostics, and candidate risk-report coverage before recurring scheduler red-line decisions.
- Active-query feedback is now validated as a scheduled red-line artifact: `active_query_feedback_report.v1` joins `active_queries.jsonl` to `oracle_pairs.jsonl`, tracks sim/real matched coverage and raw oracle result error rates, and is read by recurring scheduler red-line checks.
- Learned exploiter validation now has a combined production gate: `learned_exploiter_validation_report.v1` reads self-play `orchestrator_state.json`, embeds attack exploiter and defense anti-meta reports, checks scale/checkpoints, and prefixes nested attack/defense red lines. The current real 2-round smoke is not production-ready because attack exploiter residual quality is non-positive.
- Data-engineering validation now has a scheduled coverage gate: `data_engineering_validation_report.v1` reads league round `run_metadata.json`, validates artifact SHA-256 references, checks all five core table JSONL files, and participates in recurring scheduler red-line checks.
- Underdog residual quality is now a scheduled production gate: `underdog_residual_validation_report.v1` reads round `candidates.jsonl` and `scored_defenses.jsonl`, checks attack/defense underdog residual field coverage plus mean bonus, and participates in recurring scheduler red-line checks. The current live underdog residual smoke is not production-ready: attack underdog rows exist but mean residual bonus is 0, and defense underdog rows are absent.
- Self-play can now be validation-gated: `SelfPlayOrchestrator` can write `learned_exploiter_validation_report.v1` after each round, persist report paths/red lines in `orchestrator_state.json`, and stop early when `production_ready=true`. A live 1-round smoke verified current-round state is written before validation, so latest attack/defense checkpoints and oracle request counts are not falsely reported missing.
- Attack exploiter residual targets must compare against same-defense main attack success, not `scored_defenses.break_rate`. The latter is the max break rate over all attack roles, so it can include the exploiter/underdog candidate itself and suppress positive residual labels. After switching to a main-role baseline, the live validation-loop and early-stop smokes cleared `learned_exploiter_validation_report.v1` at small scale.
- Production readiness needs a cross-report gate, not only module-local reports. `production_readiness_report.v1` now aggregates child validation reports, checks required schema versions, read errors, child red lines, clean-report rate, and explicit `production_ready=false`, then exposes one scheduler-readable `production_ready` flag and aggregate red lines.
- Production readiness should not require manual child report paths in recurring schedules. `build_v4_training_schedule(..., production_readiness_include_scheduled_reports=True)` now auto-collects JSON outputs from validation report jobs already in the same schedule, wires producer job dependencies, and feeds those reports into the aggregate gate.
- Real calibration also needs a holdout gate, not only model fitting and drift ingestion. `real_calibration_validation_report.v1` now compares raw simulator probabilities, base calibration, and final real calibration on holdout JSONL rows, checking Brier/ECE improvements before scheduler production readiness can trust the calibrator.
- League/PSRO needs its own health gate separate from learned exploiter quality. `league_selfplay_health_report.v1` now reads round `league_state.json`, `summary.json`, `candidates.jsonl`, `scored_defenses.jsonl`, and optional `oracle_pairs.jsonl` to validate pool sizes, active retention, required role coverage, diversity clusters, payoff density, and new-strategy strength deltas before scheduler production readiness treats a self-play run as healthy.
- Active real-query dispatch needs a scheduler-readable aggregate gate, not only per-dispatch `validation_report.json`. `active_real_query_dispatch_validation_report.v1` now aggregates dispatch validation outputs and checks readable reports, real queued queries, dispatched pairs, oracle completion rate, oracle errors, and attack/defense teacher feedback completeness before recurring production readiness trusts real-query feedback.
- Active real-query dispatch feedback also needs to enter RealMetaDB, not only proposal teacher JSONL. The ingestion path now joins `real_query_pairs.jsonl` with attack/defense teacher plans, rebuilds `Observation` from the full defense plan, and writes real-meta calibration records with `source_kind=active_real_query_feedback`.
- Real calibration needs stable sample construction, not just a validator for prebuilt JSONL. `build_real_calibration_samples_from_artifacts()` now joins round/active-real labels with pre-oracle candidate predictions, prefers `predicted_score` over `surrogate_score`, embeds `RealMetaRecord`, and records source/prediction provenance so calibrator fitting can use true artifact feedback without manual JSONL assembly.
- Production readiness must account for wrapper reports with nested runtime schemas. `scripts/ingest_real_calibration.py` writes a wrapper `real_calibration_report.json` without top-level `schema_version`, while the actual `real_calibration_ingestion_summary.v1` records live under `ingestions[]`; readiness now counts those known nested schemas so required-schema checks do not falsely fail.
- Production readiness reports whether child reports are clean, but does not by itself map evidence back to the five v4 production priorities. `v4_conformance_validation_report.v1` now provides that requirement-level matrix over learned exploiter/anti-meta, full ablation feedback, real calibration holdout, active real-query dispatch, and mask/belief validation evidence. It should be treated as an acceptance gate for evidence coverage, not as a substitute for the production-scale runs themselves.
- Learned exploiter validation is now failing for a real reason in the top-50 smoke: main-role attacks often reach near-100% attack success, while exploiter/underdog attacks in the tiny candidate budget can be much weaker, producing negative anti-meta residuals. More oracle budget alone may not be enough; the attack-side exploiter candidate source or sampling policy needs stronger best-response pressure.
- Multi-role self-play with learned proposal checkpoints has a local CPU scaling hotspot. A 1-defense checkpoint round finishes, but a 3-defense/10-candidate/3-role follow-up can spend minutes at 100% CPU after completing the first backend job. This points to local generation/proposal/league-update overhead after introducing learned checkpoints, not backend throughput.
- The unsloth CUDA environment is the correct runtime for neural training/inference (`torch 2.8.0+cu128`, two RTX 4090 D visible). Running self-play with `--proposal-device cuda:0 --belief-ranker-device cuda:0` confirms checkpoint training/inference can use GPU, but round setup still spends substantial wall time in single-process CPU candidate generation/search before oracle submission. GPU usage therefore improves neural stages, not the structural generation bottleneck.
- A 3-round top50 multi-role CUDA smoke completed and cleared the earlier hard stall, but it still failed the learned exploiter gate because attack anti-meta residual stayed negative. Defense anti-meta was healthy, so the next algorithmic fix should focus on attack best-response/exploiter candidate pressure rather than defense learning.
- The "generation is strangely slow" issue was primarily a local Python generation problem, not an oracle/backend or CUDA problem. `cProfile` showed budgeted underdog random generation repeatedly failing and retrying, while each slot recomputed future-feasibility masks over sorted pools and repeated canonical hash serialization. Caching sorted pools and hashes plus budget-aware low-cost candidate filtering reduced the same learned-checkpoint local generation from minute-scale to seconds-scale.
